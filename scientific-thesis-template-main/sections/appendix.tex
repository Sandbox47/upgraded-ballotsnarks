We define groups and fields as in \cite{least_authority_moonmath_2024}.

\begin{definition}[Bilinear Pairing \cite{least_authority_moonmath_2024}]
    \label{def:bilinearPairing}
    Let $(\mathbb{G}_1, \odot_{\mathbb{G}}), (\mathbb{G}_2, \odot_{\mathbb{G}_2})$ and $(\mathbb{G}_3, \odot_{\mathbb{G}_2})$ be groups.
    Then, a bilinear pairing on these groups is a function 
    \begin{align*}
        e:\mathbb{G}_1\times\mathbb{G}_2\to\mathbb{G}_3\text{,}
    \end{align*}
    where the bilinearity property holds:
    \begin{align*}
        &\forall g_1, g_1'\in\mathbb{G}_1\forall g_2, g_2'\in\mathbb{G}_2: \\
        &\quad e(g_1\odot_{\mathbb{G}_1}g_1', g_2)=e(g_1, g_2)\odot_{\mathbb{G}_3} e(g_1',g_2) \text{ and }\\
        &\quad e(g_1, g_2\odot_{\mathbb{G}_2}g_2')=e(g_1, g_2)\odot_{\mathbb{G}_3} e(g_1,g_2') \\
    \end{align*}
    $e$ is called non-degenerate, if:
    \begin{align*}
        &\forall g_1\in\mathbb{G}_1\forall g_2\in\mathbb{G}_2:
        &\quad e(g_1, g_2)=e_{\mathbb{G}_3}\quad\Leftrightarrow\quad g_1=e_{\mathbb{G}_1} \lor g_2=e_{\mathbb{G}_2}
    \end{align*}
    Here, $e_{\mathbb{G}_1}, e_{\mathbb{G}_2}$ and $e_{\mathbb{G}_3}$ denote the neutral elements of the respective groups.
\end{definition}

As discussed in \cref{chap:elGamalEncryption}, the main computational effort and therefore also the majority of constraints are caused by the group exponentiations $g^y, h^y$ and $g^m$, when computing an \gls{eeg} ciphertext $(g^y, h^y\odot_\mathbb{G} g^m)$ where $y\probgets\{0,\dots, n-1\}$.
In \gls{ppeeg}, we reduce the effort to compute exponentiations $g^k$ with $k=\sum_{i=0}^{l-1}k_i$ by precomputing powers $\left(g^{2^i}\right)_{i=0}^{l-1}$ and employing the equivalence
\begin{align*}
    g^k=g^{\left(\sum_{i=0}^{l-1}k_i\cdot 2^i\right)}=\Pi_{i=0}^{l-1}g^{k_i\cdot 2^i}=\Pi_{i=0}^{l-1}\left(g^{2^i}\right)^{k_i}\text{.}
\end{align*}

Here, we utilized the binary representation $(k_0,\dots, k_{l-1})$ of the exponent $k$. 
We can generalize this concept to arbitrary bases for the representation of the exponent.
Assume, we have a base $base$ representation $k=\sum_{i=0}^{l'-1}k_i\cdot base^i$ (with $l'=\lceil\log_{base}(2)\cdot l\rceil$) of $k$ and want to compute $g^k$.
Additionally, we represent each digit $k_i$ of the base $base$ representation of $k$ as an array of binary indicators $(k_{i,0},\dots, k_{i,base-1})\in\{0,1\}^{base}$.
Here, we have $\forall j\in\{0,\dots, base-1\}: k_{i,j}=1\Leftrightarrow k_i=j$.
In other words, if $k_i=j$, then the entry at position $j$ in the indicator array is $1$ and all other entries are $0$.
For example, for $k=8=3\cdot 5^0+1\cdot 5^1=(3,1)=(k_0, k_1)$ we would get $(0,0,0,1,0)$ for $k_0$ and $(0,1,0,0,0)$ for $k_1$ in a representation to base $5$.

Based on this, we can employ a similar equivalence as before:
\begin{align*}
    g^k=g^{\left(\sum_{i=0}^{l'-1}k_i\cdot base^i\right)}=\Pi_{i=0}^{l'-1}g^{k_i\cdot base^i}=\Pi_{i=0}^{l'-1} \left(\Pi_{j=0}^{base-1} \left(g^{j\cdot base^i}\right)^{k_{i,j}}\right)
\end{align*}

Using this, we can extend $PrecomputedPowersFastExp$ (\cref{alg:fastExpPrecomputedPowers}) to \\$PrecomputedPowersFastExp_{base}$ as shown in \cref{alg:ppfexpArbBase} with the corresponding circuit shown in \cref{fig:circuit:ppfexpArbBase}.

\begin{Algorithmus}[h] %Use the environment only if you want to place the algorithm similar to graphics from TeX
    \caption{Algorithm $PrecomputedPowersFastExp_{base}$ used to compute $g^k$ for $g\in\mathbb{G}$ and $k\in\{0,\dots, |\mathbb{G}|-1\}$.
    Here, $k$ is provided as a representation to base $base$ of length $l$: $\left(k_{i,j}\right)_{i\in\{0,\dots, l-1\},j\in\{0,\dots, base-1\}}\in\{0,1\}^{l\times base}$ with $k=\sum_{i=0}^{l-1}\sum_{j=0}^{base-1} k_{i,j}\cdot j\cdot base^{i}$ and $\forall i\in\{0,\dots, l-1\}\exists! j:k_{i,j}=1$.
    Furthermore, $\left(g^{j\cdot base^i}\right)_{i\in\{0,\dots, l-1\}, j\in\{0,\dots, base-1\}}$ are precomputed powers of $g$.}
    \label{alg:ppfexpArbBase}
    \begin{algorithmic}
    \State \textbf{procedure} $PrecomputedPowersFastExp_{base} \Bigg(\left(g^{j\cdot base^i}\right)_{i\in\{0,\dots, l-1\}, j\in\{0,\dots, base-1\}}, $
        \Statex \hspace{9cm}%
        $\left(k_{i,j}\right)_{i\in\{0,\dots, l-1\},j\in\{0,\dots, base-1\}}\Bigg)$

                    \State $y\gets g^{j\cdot base^i}\odot_\mathbb{G} y$
                \EndIf
            \EndFor
        \EndFor
        \State \Return $y$
        \end{algorithmic}

    \end{algorithmic}

By providing $k$ in the representation to base $base$ where the individual digits are stored as indicator arrays, we reduce the number of constraint required in the circuit.
Otherwise, we would have to compute this representation in the circuit itself or use more complicated logic, both of which would require more constraints.

For base $2$, $\mathfrak{C}_{\mathbb{G}\_PPFExp_{base}}$ works very similar to $\mathfrak{C}_{\mathbb{G}\_PPFExp}$ (\cref{alg:fastExpPrecomputedPowers}).
The only differences are that we provide $k$ as a list of indicator arrays for $\mathfrak{C}_{\mathbb{G}\_PPFExp_{base}}$ whereas we provide it as a number to $\mathfrak{C}_{\mathbb{G}\_PPFExp}$ and that we switch the order of the group law application and the selection of the result in each iteration.

Next, we want to determine the optimal base $base$ that yields the smallest constraint count for $\mathfrak{C}_{\mathbb{G}\_PPFExp_{base}}$.
For this purpose, we introduce a cost function $cost_\mathbb{G}(base)$ which calculates the number of constraints per "Bit-Equivalent"\ $\log_{base}(2)$.
We employ this normalization, since a number that can be represented with $255$ bits can also be represented with $\Big\lceil \frac{255}{\log_2(base)}\Big\rceil$ digits in base $base$.
For instance, we need $\Big\lceil \frac{255}{\log_2(5)}\Big\rceil=110$ digits to represent a $255$ bit number in base $5$.
Furthermore, the cost is influenced by the number of constraint needed for the group law $\mathfrak{C}_\mathbb{G}\_\odot$ and the group switch case $\mathfrak{C}_{\mathbb{G}\_Switch\text{-}Case(base)}$.
We denote the constraint counts for the individual circuits by $\mathcal{N}(\mathfrak{C}_\mathbb{G}\_\odot)$ and $\mathcal{N}(\mathfrak{C}_{\mathbb{G}\_Switch\text{-}Case(base)})$.
Putting everything together, we get 
\begin{align*}
    cost_\mathbb{G}(base) = \frac{\mathcal{N}(\mathfrak{C}_{\mathbb{G}\_Switch\text{-}Case(base)}) + \mathcal{N}(\mathfrak{C}_\mathbb{G}\_\odot)}{log_2(base)}\text{.}
\end{align*}

As before, we use Twisted Edwards curves as the group $\mathbb{G}$ since the group law $\mathfrak{C}_\mathbb{G}\_\odot$ requires fewer constraints compared to the group law for Montgomery Curves.
Additionally, Twisted Edwards curves have only two coordinates whereas Montgomery Curves have three coordinates, which means that $\mathfrak{C}_{\mathbb{G}\_Switch\text{-}Case(base)}$ uses fewer constraints as well.
Specifically, with $\mathfrak{C}_{TE_{a,d}(\mathbb{F})\_Switch\text{-}Case(c)}$ as presented in \cref{fig:circuit:teSwitchCase}, we get $\mathcal{N}(\mathfrak{C}_{TE_{a,d}(\mathbb{F})\_Switch\text{-}Case(base)})=2\cdot(base-1)$ and $\mathcal{N}(\mathfrak{C}_{TE_{a,d}(\mathbb{F})\_\odot})=7$.
Then, \begin{align*}
    cost_{TE_{a,d}(\mathbb{F})}(base)= \frac{2\cdot(base-1) + 7}{\log_2(base)}\text{.}
\end{align*}

We can conclude that $base=5$ is optimal for $\mathbb{G}=TE_{a,d}(\mathbb{F})$.
Note that choosing base $base$ requires us to precompute $\frac{base-1}{\log_2(base)}$ times as many powers of $g$ as in the circuit $\mathfrak{C}_{\mathbb{G}\_PPFExp}$.
For our choice $base=5$ this means that we need to precompute $\frac{4}{\log_2(5)}=1.72$ times as many powers of $g$ as before.
However, this is a sensible tradeoff since these computations are typically not executed by the voter but rather the election organizers and are not part of the ballot validity proofs.
Furthermore, as before, we can share the precomputed powers of $g$ among all instantiations of $\mathfrak{C}_{\mathbb{G}\_PPFExp_{base}}$ in a ballot validity circuit.

While we did implement $\mathfrak{C}_{TE_{a,d}(\mathbb{F})\_PPFExp_{base}}$ in Circom, it would be interesting to build a new version of \gls{ppeeg} using this.
Furthermore, one could integrate this with the rest of the ballot validity circuits and implement the corresponding SageMath component which generates the required powers of $g$ and the indicator based representations of the exponent $k$.
After obtaining benchmarks for this new implementation, one could compare the results to the results we obtained and examine whether this variant yields the expected constraint reductions in the full ballot validity circuits.

\subsection{$If\text{-}Else$}
\label{section:appendix:ifElse}
Branching is very important for most use cases.
Often, the result of a computation depends on a condition.
For this purpose, we use \begin{align*}
    r_{If\text{-}Else(l)}:&\mathbb{F}^l\times \mathbb{F}^l\times\mathbb{F}\to\mathbb{F}^l, (v_{else},v_{if}, cond)\mapsto \begin{cases}
        v_{else},\quad \text{if } cond=0\\
        v_{if},\quad\text{if } cond=1\\
        undefined,\quad\text{otherwise}
    \end{cases} \\
    &\quad\text{with }\begin{matrix}
        v_{else} & = (v_0^{else}, \dots, v_{l-1}^{else})\\
        v_{if} & = (v_0^{if}, \dots, v_{l-1}^{if})
    \end{matrix}
\end{align*}
and the corresponding circuit $\mathfrak{C}_{If\text{-}Else(l)}$ \cref{fig:circuit:ifElse}.

\subsection{$Switch\text{-}Case$}
\label{section:appendix:switchCase}
Branching is very important for most use cases.
Often, the result of a computation depends on a condition that can take several different values with a default value if none of the allowed values are chosen.
For this purpose, we use \begin{align*}
    &r_{Switch\text{-}Case(c)}:\mathbb{F}^c\times \mathbb{F}^{c-1}, \\
    &\quad(v, cond)\mapsto v_{res}:=\begin{cases}
        v_{i},\quad \text{if } cond_i=1 \land \forall j\in\{0,\dots,c-2\}\setminus\{i\}: cond_j=0\\
        v_{def},\quad\text{if } \forall i\in\{0,\dots, c-2\}:cond_i=0\\
        undefined,\quad\text{otherwise}
    \end{cases} \\
    &\quad\quad\text{with }\quad\begin{matrix*}[l]
        v &= (v_0, \dots, v_{c-2}, v_{def})\\
        cond &= (cond_0, \dots, cond_{c-2})
    \end{matrix*}
\end{align*}
and the corresponding circuit $\mathfrak{C}_{Switch\text{-}Case(c)}$ \cref{fig:circuit:switchCase}.

\subsection{$Safe\text{-}Division$}
\label{section:appendix:safeDivision}
We introduced the circuit $\mathfrak{C}_{Division}$ to compute $\frac{x}{y}$ for two inputs $x$ and $y$ earlier (\cref{fig:circuit:division}).
However, this is problematic if $y=0$.
In such cases, we typically do not use the result of the division circuit, for instance because it is the input for a not enabled branch of $\mathfrak{C}_{Switch\text{-}Case(c)}$.
Despite this, we need to be able to handle the case $y=0$ since, in a circuit, all computations are always executed.
For this purpose, we use \begin{align*}
    r_{Division\text{-}Safe}:\mathbb{F}^2\to\mathbb{F}, (x,y)\mapsto res:=\begin{cases}
        x,\quad \text{if } y=0\\
        \frac{x}{y},\quad \text{otherwise}
    \end{cases}
\end{align*}
and the corresponding circuit $\mathfrak{C}_{Safe\text{-}Division}$ \cref{fig:circuit:safeDivision}.

\subsection{Comparisons}
\label{section:appendix:comparisons}
For a lot of use cases, we need to determine the ordering of values relative to each other.
In our implementation, we mainly use this in the circuit $\mathfrak{C}_C^{Vot}$ asserting choice space membership for some election types.

For this purpose, we use \begin{align*}
    & r_{Is\text{-}Less}:\mathbb{F}^2\to\mathbb{F}, (x,test)\mapsto\begin{cases}
        1,\quad\text{if } x < test\\
        0,\quad\text{otherwise}
    \end{cases}\\
    & r_{Is\text{-}Leq}:\mathbb{F}^2\to\mathbb{F}, (x,test)\mapsto\begin{cases}
        1,\quad\text{if } x \leq test\\
        0,\quad\text{otherwise}
    \end{cases}\\
    & r_{Is\text{-}Greater}:\mathbb{F}^2\to\mathbb{F}, (x,test)\mapsto\begin{cases}
        1,\quad\text{if } x > test\\
        0,\quad\text{otherwise}
    \end{cases}\\
    & r_{Is\text{-}Geq}:\mathbb{F}^2\to\mathbb{F}, (x,test)\mapsto\begin{cases}
        1,\quad\text{if } x \geq test\\
        0,\quad\text{otherwise}
    \end{cases}\\
\end{align*}
and the corresponding circuits $\mathfrak{C}_{Is\text{-}Less}$, $\mathfrak{C}_{Is\text{-}Leq}$, $\mathfrak{C}_{Is\text{-}Greater}$ and $\mathfrak{C}_{Is\text{-}Geq}$ (\cref{fig:circuit:comparisons}).
In our implementation, we use the templates \verb|LessThan(n)|, \verb|LessEqThan(n)|, \verb|GreaterThan(n)| and \verb|GreaterEqThan(n)| from the file \verb|comparators.circom| in the circomlib library \cite{jordy_baylina_iden3circomlib_2025}.
Note that the parameter $n$ denotes the number of bits the binary representations of $x$ and $test$ can have at most. 
We use a sufficiently large $n$ everywhere in our implementation and abstract away from this detail.

\subsection{Counting}
\label{section:appendix:counting}
In some use cases, we need to count the number of times a value with some specific property occurs in a list of values.
In our implementation, we make use of this in the circuit $\mathfrak{C}_C^{Vot}$ asserting choice space membership for some election types.

For this purpose, we use \begin{align*}
    & r_{Count\text{-}Equal(l)}:\mathbb{F}^l\times\mathbb{F}\to\mathbb{F}, ((v_0,\dots, v_{l-1}), test)\mapsto count:=|\{i|v_i=test\}|\\
    & r_{Count\text{-}Less(l)}:\mathbb{F}^l\times\mathbb{F}\to\mathbb{F}, ((v_0,\dots, v_{l-1}), test)\mapsto count:=|\{i|v_i<test\}|\\
    & r_{Count\text{-}Greater(l)}:\mathbb{F}^l\times\mathbb{F}\to\mathbb{F}, ((v_0,\dots, v_{l-1}), test)\mapsto count:=|\{i|v_i>test\}|\\
\end{align*}
and the corresponding circuits $\mathfrak{C}_{Count\text{-}Equal}$, $\mathfrak{C}_{Count\text{-}Less}$ and $\mathfrak{C}_{Count\text{-}Greater}$ (\cref{fig:circuit:counting}).
Here, $r_{Count\text{-}Equal(l)}$, $r_{Count\text{-}Less(l)}$ and $r_{Count\text{-}Greater(l)}$ count how many values in a list of length $l$ have the desired relation with a test value $test$.
