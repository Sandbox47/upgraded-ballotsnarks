\chapter{Software}
\label{chap:software}

We use software to implement circuits, create proofs of correct circuit execution and to provide the correct inputs to the ballot validity assertion circuits.
The accompanying implementation contains all source code we wrote for this work.

\section{Circom}

In \cref{chap:gpzkp} we introduced \glspl{zkp} of correct circuit execution over arithmetic circuits.
These proofs constitute a form of \gls{gpzkp} since we can derive a \gls{zkp} for any statement expressible inside a circuit.

Previously, we defined circuits to assert ballot validity in our voting schemes (\cref{chap:ourVotingSchemes}).
So far, we represented these circuits visually.
However, to implement circuits and automatically derive a proof of correct circuit execution, we need to express them in a Circuit Language.
In contrast to Huber et al. \cite{huber_zk-snarks_2025}, who used libsnark \cite{madars_virza_scipr-lablibsnark_2025}, we choose Circom \cite{rodriguez_circom_2025} as the Circuit Language.
Circom is advantageous when compared to libsnark for multiple reasons:

The main benefit is, that Circom is used purely to define circuits and to convert them to the corresponding \gls{r1cs} representation. 
Therefore, we can choose any \gls{zps} that uses \glspl{r1cs} for the proofs of correct circuit execution.
As we did in this thesis, Circom is most commonly used with the backend library snarkjs \cite{jordi_baylina_iden3snarkjs_2025} and the \gls{zps} \gls{g16snark} \cite{groth_size_2016}.
However, snarkjs also supports the \glspl{zps} PLONK \cite{gabizon_plonk_2019} and FFLONK \cite{gabizon_fflonk_2021}.
Moreover, using other backends such as arkworks \cite{yuncong_hu_arkworks_2025} with other \gls{zps} such as marlin \cite{chiesa_marlin_2019,pratyush_mishra_arkworks-rsmarlin_2025} and Gemini \cite{bootle_gemini_2022} is also possible.
libsnark on the other hand is an End-to-End \gls{zkp} system which implements \glspl{g16snark} and \glspl{snark} based on the approach by Ben-Sasson et al. \cite{ben-sasson_succinct_2013}.

Additionally, Circom provides higher-level primitives and easier syntax for defining circuits, leading to implementations that are easier to create, use and maintain than in libsnark.
Moreover, Circom is still being maintained whereas libsnark does not support recent proof systems and there are no new features being added to it.

\subsection{Syntax}
The syntax of Circom is similar to that of JavaScript.
We explain the syntax using the circuit $\mathfrak{C}_{3\text{-}vot}$ \cref{fig:circuit:exampleVoting} as an example.

We define a circuit with the keyword \verb|template| and use a notation similar to JavaScript methods.
Variables that participate in constraints are denoted with \verb|signal|.\footnote{
    \label{footnote:circomSignalElimination}
    When computing the \gls{r1cs} corresponding to a circuit, Circom tries to minimize the number of equations (i.e., constraints).
    In this process, some intermediate signals might get eliminated and are therefore not part of the \gls{r1cs}.
}
Additionally, inputs and outputs are denoted with \verb|input| and \verb|output| respectively.
In order to assign a value or variable \verb|v| to a variable \verb|variable|, we write \verb|variable <-- v|.
However, to enforce that \verb|variable| actually takes the value \verb|v|, we need to add this constraint to the \gls{r1cs} corresponding to the circuit.
We denote this by \verb|variable === v|. 
We can also combine assigning and constraining in one notation \verb|variable <== v|.

Lastly, the basic operations $+$ and $\cdot$ are denoted by \verb|+| and \verb|*| and \verb|-| is supported as well.
Here, \verb|a-b| denotes $a+(-1)\cdot b$.

With these basics in mind, we can implement the circuit $\mathfrak{C}_{3\text{-}vot}$ as the Circom template \verb|threeVot| (\cref{fig:circom:naiveThreeVot}).

\begin{figure}[h]
    \centering
    \captionsetup{type=listing}
    \begin{lstlisting}[language=Circom]
pragma circom 2.2.1;

template threeVot() {
    input signal S_1_in;
    input signal S_2_in;
    input signal S_3_in;

    output signal S_1_out;
    output signal S_2_out;
    output signal S_3_out;
    output signal S_4_out;

    S_1_out <== S_1_in*(1-S_1_in);
    S_2_out <== S_2_in*(1-S_2_in);
    S_3_out <== S_3_in*(1-S_3_in);

    S_4_out <== (S_1_in + S_2_in + S_3_in)*(1 - S_1_in - S_2_in - S_3_in);
}\end{lstlisting}
    \caption{Naive implementation of $\mathfrak{C}_{3\text{-}vot}$ in Circom.}
    \label{fig:circom:naiveThreeVot}
\end{figure}

As introduced earlier (\cref{chap:gpzkp}), $\mathfrak{C}_{3\text{-}vot}$ is used to indicate whether a ballot $b=(S_1^{in}, S_2^{in}, S_3^{in})$ is in the choice space $C=\{(0,0,0), (0,0,1), (0,1,0), (1,0,0)\}$.
If this is the case, the circuit will produce the outputs $(S_1^{out}, S_2^{out}, S_3^{out}, S_4^{out})=(0,0,0,0)$.
To use the circuit to assert that a ballot is in $C$, we need to enforce $(S_1^{out}, S_2^{out}, S_3^{out}, S_4^{out})\mbeq(0,0,0,0)$.
We call this modified circuit $\mathfrak{C}_{Assert\text{-}3\text{-}vot}$ (\cref{fig:circuit:exampleAssertThreeVot}) and can define this in Circom as \verb|assertThreeVot| (\cref{fig:circom:assertThreeVot}).
However, to explain the core concepts of Circom, we continue with $\mathfrak{C}_{3\text{-}vot}$ for the rest of this section.

\begin{figure}[h]
    \centering
    \captionsetup{type=listing}
    \begin{lstlisting}[language=Circom]
pragma circom 2.2.1;

template assertThreeVot() {
    input signal S_1_in;
    input signal S_2_in;
    input signal S_3_in;

    0 === S_1_in*(1-S_1_in);
    0 === S_2_in*(1-S_2_in);
    0 === S_3_in*(1-S_3_in);

    0 === (S_1_in + S_2_in + S_3_in)*(1 - S_1_in - S_2_in - S_3_in);
}\end{lstlisting}
    \caption{Implementation of $\mathfrak{C}_{Assert\text{-}3\text{-}vot}$ in Circom.}
    \label{fig:circom:assertThreeVot}
\end{figure}

In addition to the circuit specific constructs, Circom supports most of the basic programming constructs from JavaScript such as functions, branching and loops.
Furthermore, we can use \verb|var variable| for temporary variables whose computation does not add any constraints.\footnote{
    \label{footnote:tempVariablesInConstraints}
    It is possible to use temporary variables in constraints as long as no additional constraints are needed to compute them.
    We see this in the next example in \cref{fig:circom:threeVot}.
}
Arrays are supported for both \verb|signal| and \verb|var|.
Using these new constructs, we can simplify \verb|threeVot| as shown in \cref{fig:circom:threeVot}.

\begin{figure}[h]
    \centering
    \captionsetup{type=listing}
    \begin{lstlisting}[language=Circom]
pragma circom 2.2.1;

template threeVot() {
    input signal S_in[3];

    output signal S_out[4];

    var sum = 0;
    for(var i = 0; i < 3; i++) {
        S_out[i] <== S_in[i]*(1-S_in[i]);
        sum += S_in[i];
    }

    S_out[3] <== sum * (1-sum);
}\end{lstlisting}
    \caption{Implementation of $\mathfrak{C}_{3\text{-}vot}$ in Circom using basic JavaScript constructs.}
    \label{fig:circom:threeVot}
\end{figure}

To make the design of complicated circuits possible, we need a way of using circuits inside other circuits.
In the example of $\mathfrak{C}_{3\text{-}vot}$, we used a circuit $\mathfrak{C}_{Is\text{-}Bit}$ (\cref{fig:circuit:bitModularity}) to check whether all individual votes and the sum of all votes are bits.
Circom supports this concept of circuit modularity with the keyword \verb|component| which is used to denote the instantiation of a template inside another one.
Using this mechanism, we can extract the circuit $\mathfrak{C}_{Is\text{-}Bit}$ into a separate template \verb|isBit| (\cref{fig:circom:threeVotIsBit}).
Then, \verb|threeVot| can be implemented as shown in \cref{fig:circom:threeVotIsBit}.

\begin{figure}[h]
    \centering
    \captionsetup{type=listing}
    \begin{lstlisting}[language=Circom]
pragma circom 2.2.1;

template isBit() {
    input signal in;

    output signal out;

    out <== in*(1-in);
}

template threeVot() {
    input signal S_in[3];

    output signal S_out[4];

    component isBit[4];
    isBit[3] = isBit();

    var sum = 0;
    for(var i = 0; i < 3; i++) {
        isBit[i] = isBit();
        isBit[i].in <== S_in[i];
        S_out[i] <== isBit[i].out;
        sum += S_in[i];
    }

    isBit[3].in <== sum;
    S_out[3] <== isBit[3].out;
}\end{lstlisting}
    \caption{Implementation of $\mathfrak{C}_{Is\text{-}Bit}$ and $\mathfrak{C}_{3\text{-}vot}$ utilizing the $\mathfrak{C}_{Is\text{-}Bit}$ (\cref{fig:circuit:bitModularity}) template in Circom.}
    \label{fig:circom:threeVotIsBit}
\end{figure}


Assume, we want to compute whether a ballot of $n$ votes rather than just $3$ votes is valid.
The circuit $\mathfrak{C}_{n\text{-}vot}$ to compute this is defined analogously to $\mathfrak{C}_{3\text{-}vot}$.
In Circom, this concept is captured with parameters of templates.
By parametrizing a new template \verb|vot| with \verb|n|, we can implement $\mathfrak{C}_{n\text{-}vot}$ in Circom (\cref{fig:circom:nVot}).

\begin{figure}[h]
    \centering
    \captionsetup{type=listing}
    \begin{lstlisting}[language=Circom]
pragma circom 2.2.1;

template vot(n) {
    input signal S_in[n];

    output signal S_out[n+1];

    component isBit[n+1];
    isBit[n] = isBit();

    var sum = 0;
    for(var i = 0; i < n; i++) {
        isBit[i] = isBit();
        isBit[i].in <== S_in[i];
        S_out[i] <== isBit[i].out;
        sum += S_in[i];
    }

    isBit[n].in <== sum;
    S_out[n] <== isBit[n].out;
}\end{lstlisting}
    \caption{Implementation of $\mathfrak{C}_{n\text{-}vot}$ in Circom.}
    \label{fig:circom:nVot}
\end{figure}

\subsection{Instance and Witness}
Strictly speaking, the templates \verb|threeVot| and \verb|vot| are not circuits yet but rather \glqq templates\grqq\ of circuits that need to be instantiated to create circuits.
In Circom, we can do that by specifying the \verb|main| component.
For parametrized circuits such as \verb|vot|, we also need to assign values to all parameters since the circuit typically has a different size or layout depending on them.
By assigning $3$ to \verb|n|, we can create an instantiation of \verb|vot| that is equivalent to \verb|threeVot|:

\begin{lstlisting}[language=Circom]
    component main = vot(3);
\end{lstlisting}

In \cref{section:proofSchemes:instanceAndWitness}, we discussed how we can decompose an assignment to a circuit into a private witness and a public instance.
In Circom, all inputs to a circuit are part of the witness by default, but can be specified to be part of the instance by including them in a \verb|public| list when instantiating the circuit.
All outputs of a circuit in Circom are part of the instance and cannot be made part of the witness.
Therefore, the decomposition into instance and witness of $\mathfrak{C}_{3\text{-}vot}$ (\cref{fig:circuit:instanceWitnessRelationThreeVot}) where all inputs are part of the witness is the default in Circom.

Assume, we wanted to implement a different decomposition, where the first of the submitted votes is also revealed and part of the instance.
Then, we can implement this in Circom as follows:

\begin{lstlisting}[language=Circom]
    component main {public [S_in[0]]}= vot(3);
\end{lstlisting}

\subsection{Assignments and Circom \glqq Witness\grqq\ Generation}
To create a proof of correct circuit execution, we first need to convert the circuit to a corresponding \gls{r1cs} and assign values to all input, output and intermediate signals.
Values for the input signals are provided in an \verb|input.json| file.
For 
\begin{lstlisting}[language=Circom]
    component main = vot(3);
\end{lstlisting}
and an exemplary ballot $b=(S_1^{in}, S_2^{in}, S_3^{in})=(1,0,1)$, we need to create the \verb|input.json| file as follows:

\begin{lstlisting}[language=json]
{
    "S_in" : [1, 0, 1]
}\end{lstlisting}

Additionally, we need to calculate values for the intermediate and output signals such that all constraints of the circuit are fulfilled \cite{rodriguez_circom_2025}.
For this purpose, Circom provides a so-called \glqq Witness\grqq\ Generation procedure which assigns appropriate values to these signals.\footnote{
    \label{footnote:circomWitness}
    Note that the term \glqq Witness\grqq\ in the context of Circom \glqq Witness\grqq\ Generation has a very different meaning compared to the term witness describing the private parts of an assignment to a circuit.
}
In the example \verb|component main = vot(3)|, the signals that still need to be assigned are the outputs \verb|S_out|.
Here, the Circom \glqq Witness\grqq\ generation assigns \verb|S_out = [0,0,0,-2]| to fulfill all constraints.\cref{footnote:fieldNegativeNumbers}
Since the sum of all votes is $1+1=2$, $S_{out}^4=-2\neq 0$, the ballot $(1,0,1)$ is not part of the choice space in our example voting scheme from \cref{chap:votingschemes}.

\SaveVerb{compO1}|--O1|
\SaveVerb{compO2}|--O2|
Assume, we have a \verb|vot.circom| file with \verb|component main = vot(3)|.
Then, we can generate the corresponding \gls{r1cs} and create the full assignment described previously as in \cref{fig:shell:circomExample}.\footnote{
    \label{footnote:optimizationOption}
    With the command \Verb|circom vot.circom --r1cs --wasm|, we compile our circuits with the default optimization option \Verb|--O1|.
    For the zk-\gls{snark} of our choice, Groth16, the optimization option \Verb|--O2| can be used safely and is arguably the better choice as it simplifies constraints as much possible.
    In that case, the compilation command would be \Verb|circom vot.circom --r1cs --wasm --O2|.
    However, we used the default optimization \Verb|--O1| for our benchmarks.
}
After this, the instantiated \gls{r1cs} is saved in the \verb|witness.wtns| file.

In the previous example, Circom \glqq Witness\grqq\ generation is equivalent to circuit execution.
However, in general, Circom \glqq Witness\grqq generation is more powerful since it is used to \textit{find} values such that the constraints are satisfied.
We can make use of this for the utility circuit $\mathfrak{C}_{Division}$ (\cref{fig:circuit:division}) which computes $\frac{x}{y}$ for inputs $x$ and $y$.
The corresponding Circom implementation is shown in \cref{fig:circom:division}.
Here, Circom \glqq Witness\grqq\ generation is employed to find a value $out$ such that $out\cdot y=x$ which is equivalent to $out=\frac{x}{y}$.
Achieving the same result with pure circuit execution would be more complicated.

\section{snarkjs}
\label{section:software:snarkjs}

snarkjs \cite{jordi_baylina_iden3snarkjs_2025} is used to create \glspl{zkp} for \glspl{r1cs}.
We choose snarkjs since it is well integrated with Circom.
Moreover, snarkjs supports multiple different \glspl{zps} (Groth16 \cite{groth_size_2016}, PLONK \cite{gabizon_plonk_2019} and FFLONK\cite{gabizon_fflonk_2021}) and instantiation of these \glspl{zps} with multiple different elliptic curves such as \verb|BN254|\cite{jonathan_wang_bn254_2024} and \verb|BLS12-381|\cite{ben_edgington_bls12-381_2023}.
This makes it a versatile choice for many different use cases.

In our implementation, we used the \gls{zps} Groth16 operating over the elliptic curve \verb|BN254|.

The three different algorithms $Setup$, $Prove$ and $Verify$ of \glspl{g16snark} (\cref{section:proofSchemes:groth16}) are implemented separately in snarkjs.

\subsubsection{Setup}
\label{section:software:snarkjsSetup}
In $Setup$ (\cref{alg:setupG16}), we compute $CRS=(CRS_{\mathbb{G}_1}, CRS_{\mathbb{G}_2})$ for a \gls{r1cs} $R1CS_r$ based on a randomly chosen simulation trapdoor $st=(\alpha, \beta, \gamma, \delta, \tau)\in\mathbb{F}_q^5$.
Note that all parameters 
\begin{align*}
    \textcolor{olive}{g_2^\beta}, \textcolor{green}{g_2^\gamma},\textcolor{violet}{g_2^\delta},\left(\textcolor{pink}{g_2^{\tau^j}}\right)_{j=0}^{k-1}
\end{align*}
in $CRS_{\mathbb{G}_2}$ do not depend on $R1CS_r$ or only depend on the number of constraints $k$ in $R1CS_r$ but not on the constraints themselves.
Similarly, in $CRS_{\mathbb{G}_1}$ the parameters 
\begin{align*}
    \textcolor{red}{g_1^\alpha},\textcolor{blue}{g_1^\beta},\textcolor{lime}{g_1^\delta},\left(\textcolor{teal}{g_1^{\tau^j}}\right)_{j=0}^{k-1}
\end{align*}
do not depend on $R1CS$ or only depend on the number of constraints $k$.

Therefore, we can split up the generation of $CRS$ into a part $CRS\text{-}Universal$ and a part $CRS\text{-}R1CS_r$ that can be combined to the full $CRS$.
Here, $CRS\text{-}Universal$ is independent of the circuit and can be used for any circuits up to some fixed maximum amount of constraints $k_{max}$.
$CRS\text{-}R1CS_r$ on the other hand, is specific to the circuit.

In our voting schemes, we want to create \glspl{zkp} for many different election types and different candidate counts.
Therefore, we can generate $CRS\text{-}Universal$ once for a number of constraints $k_{max}$ sufficient for all scenarios.
Then, for every circuit, we only need to generate $CRS\text{-}R1CS_r$ and combine this with $CRS\text{-}Universal$ to obtain the full $CRS$ for this circuit.
This can be very efficient compared to generating the full $CRS$ from scratch every time.

\SaveVerb{ptau}|powersoftau.ptau|
In snarkjs, $CRS\text{-}Universal$ is represented by a \verb|powersoftau.ptau| file, which is typically generated using a \gls{mpc} protocol with several contributions from different sources such as random beacons \cite{nicolas_gailly_league_2025,popper_powers--tau_2024,bowe_scalable_2017}.\footnote{
    For our benchmarks, we used the pregenerated \protect\UseVerb{ptau} files from \cite{jordi_baylina_iden3snarkjs_2025} which have been generated with many contributions from different sources.
}
To create $CRS\text{-}R1CS_r$, snarkjs uses \verb|powersoftau.ptau| and the $R1CS$ of a circuit in the form of a \verb|circuit.r1cs| file.
Here, we typically perform another \gls{mpc} to contribute randomness from different sources to the $CRS\text{-}R1CS_r$.\footnote{
    For our benchmarks, we created the full $CRS$ directly from $CRS\text{-}Universal$ without using any additional contributions.
    Note that this approach is insecure and should not be used in practice.
    Therefore, we also implement the option to contribute randomness from the NIST 2.0 random beacon \cite{computer_security_division_nist_2019}.
    However, this is disabled at the moment.
}
The final $CRS$ is then saved in a \verb|key.zkey| file and a subset of the $CRS$ needed to verify proofs is exported as \verb|verification_key.json|.

\subsubsection{Prove}
\label{section:software:snarkjsProve}
In order to create a Groth16 proof for a circuit, we require an instantiation of the \gls{r1cs} for this circuit as a \verb|witness.wtns| file and an appropriate $CRS$ as a \verb|key.zkey| file.
Then, we can create the proof as in the $Prove$ algorithm (\cref{alg:proveG16}) using the \verb|groth16 prove| command.
This creates two files \verb|proof.json| and \verb|public.json| containing the proof itself and the instance, i.e., an assignment to all public signals of the circuit.

\subsubsection{Verify}
To verify a Groth16 proof for a circuit, we require a specific subset of the $CRS$ as \verb|verification_key.json|, the proof \verb|proof.json| as generated by $Prove$ \cref{alg:proveG16} and an instance as \verb|public.json|.
Then, we can verify the proof as in the $Verify$ algorithm (\cref{alg:verifyG16}) using the \verb|groth16 verify| command.

\section{SageMath}
When proving ballot validity, a voter needs to submit the ballot $b$ they chose to a \gls{zps}.
However, to construct the proof, we also require the corresponding encrypted ballot $b'$ as well as the public key $pk$ and the randomness $aux^{Enc}$ that was used in the computation of $b'$.
In case of ranking based voting methods, we sometimes also need the private ranking $aux^{Vot}=r$ chosen by the voter to construct the ballot.

In a real world voting system, a voter should only have to choose the ballot $b$ or, in case of ranking based voting schemes, the ranking $r$.
The rest of the inputs required for the \gls{zkp} of ballot validity should be generated automatically.
In our implementation, we employ SageMath \cite{matthias_koppe_sagemath_2025} for this purpose.
SageMath is a good choice since the encryption of ballots is defined utilizing finite fields and elliptic curves which are already implemented efficiently in SageMath.
However, SageMath is not designed for cryptographic purposes and might be susceptible to generic attack strategies such as Side-Channel attacks.
Therefore, our implementations in SageMath should not be used in practice without thorough security analysis beforehand.
%Additionally, SageMath is based on Python which makes it easy to integrate into 

In most election types, the voters directly choose their ballots as part of the voting procedure.
However, in ranking based election types, the voters merely choose a ranking $aux^{Vot}=r$ from which the ballot $b$ is computed.
Therefore, we provide this functionality of computing a ballot from a ranking for the election types Pointlist-Borda, \gls{bts} and Condorcet in SageMath.

To compute the encryption of a singular ballot entry $b_i$ in the encryption scheme $EEG(\mathbb{G})$ or $PPEEG(\mathbb{G})$, we need a randomness $aux^{Enc}_i$ in the range $0,\dots, |\mathbb{G}|-1$.
For this purpose, we provide a SageMath implementation as well.

Using the ballot $b$ and the randomness $aux^{Enc}$, we employ \gls{eeg} or \gls{ppeeg} to check the correctness of encryption based on circuits, as shown in \cref{chap:elGamalEncryption}.
As discussed previously, we either use $EEG(M_{A,B}(\mathbb{F}P^2))$ or $PPEEG(TE_{a,d}(\mathbb{F}))$.
To enable encryption of $b$, we now need to generate a public key $pk=(M_{A,B}(\mathbb{F}P^2), |M_{A,B}(\mathbb{F}P^2)|, g, h)$ or $pk=\left(TE_{a,d}(\mathbb{F}), n, \left(g^{2^i}\right)_{i=0}^{\lfloor \log_2(n)\rfloor}, \left(h^{2^i}\right)_{i=0}^{\lfloor \log_2(n)\rfloor}\right)$ with $n:=|TE_{a,d}(\mathbb{F})|$ respectively.
Finally, we need to compute the encrypted ballot $b'$ as either $EEG(M_{A,B}(\mathbb{F}P^2)).Enc^{aux^{Enc}}(b, pk)$ or $PPEEG(TE_{a,d}(\mathbb{F})).Enc^{aux^{Enc}}(b, pk)$.
SageMath has built-in support of elliptic curve arithmetic, but only for Short Weierstrass curves.
Recall that for every Montgomery curve there is an isomorphic Twisted Edwards curve and vice versa and that for every Montgomery curve there is an isomorphic Short Weierstrass curve.
Therefore, we can make use of the built-in Short Weierstrass curves of SageMath by converting elliptic curve points to and from this representation.

More formally, let $(TE_{a,d}(\mathbb{F}), \odot_{TE})$ be a Twisted Edwards curve, let $(M_{A,B}(\mathbb{F}P^2), \odot_M)$ be the isomorphic Montgomery curve (\cref{theorem:TEMIsomorphism}) and let $(\mathbb{S}, \odot_\mathbb{S})$ be the isomorphic Short Weierstrass curve.
Then, we can compute $g_{M,1}\odot_{M} g_{M, 2}$ with $g_{M,1}, g_{M, 2}\in M_{A,B}(\mathbb{F}P^2)$ using the following procedure: 
First, we convert $g_{M, 1}$ and $g_{M, 2}$ to the corresponding Short Weierstrass points $g_{\mathbb{S}, 1}, g_{\mathbb{S}, 2}\in\mathbb{S}$.
Then, we compute $g_{\mathbb{S}, res}:=g_{\mathbb{S}, 1}\odot_\mathbb{S} g_{\mathbb{S}, 2}$ using the built-in implementation from SageMath and convert $g_{\mathbb{S}, res}$ back to the corresponding Montgomery point $g_{M, res}$.

Similarly, we compute $g_{TE,1}\odot_{TE} g_{TE, 2}$ with $g_{TE,1}, g_{TE, 2}\in TE_{a,d}(\mathbb{F})$:
First, we convert the points to the corresponding Montgomery points $g_{M, 1}, g_{M, 2}$ and from there to the corresponding Short Weierstrass points $g_{\mathbb{S}, 1}, g_{\mathbb{S}, 2}\in\mathbb{S}$.
Then, we compute $g_{\mathbb{S}, res}:=g_{\mathbb{S}, 1}\odot_\mathbb{S} g_{\mathbb{S}, 2}$ using the built-in implementation from SageMath and convert $g_{\mathbb{S}, res}$ to $g_{M, res}$ and from there to the corresponding Twisted Edwards point $TE_{M, res}$.
We can also visualize this procedure as illustrated in \cref{fig:sage:groupLaw}.

With SageMath, we can then compute both $pk$ and $b'$ utilizing the conversion mechanism and the built-in Short Weierstrass arithmetic.

In addition to the use cases above, we employ SageMath here to generate random ballots in the different election types we consider.
Then, we use SageMath to create the \verb|input.json| files holding the inputs for our implementation of $\mathfrak{C}_C^{Ballot\text{-}Validity}$ in Circom.
